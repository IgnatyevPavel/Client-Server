### 3. что такое REST API
**REST API** - это архитектурный подход, который устанавливает ограничения для API (как API должен быть устроен и какие функции должен поддерживать). Это позволяет стандартизировать работу программных интерфейсов, сделать их более удобными и производительными.
___
### 4. требования к REST API
Есть шесть принципов, на которых строится REST: 

+ Клиент-серверная модель: методология REST следует принципу разделения клиента и сервера. 

+ Отсутствие состояния: Каждый запрос от клиента должен содержать всю необходимую информацию для его обработки на сервере. Сервер не должен хранить информацию о предыдущих запросах клиента. 

+ Кэширование: Сервер может указывать, может ли клиент кэшировать ответы на запросы, для уменьшения нагрузки на сервер и улучшения производительности.

+ Единообразный интерфейс: Используйте унифицированные URL и форматы данных для обеспечения согласованности. REST API должен иметь унифицированный интерфейс, который состоит из четырех основных компонентов:

+ Многоуровневая система: Разделите функции на разные уровни для улучшения масштабируемости и гибкости. REST API может быть построено как многоуровневая система, где промежуточные серверы могут обрабатывать запросы перед их достижением конечного сервера. Это позволяет улучшить масштабируемость и надежность системы.

+ Код по требованию: Выполняйте клиентский код при необходимости для расширения функциональности. Код по требованию: сервер может расширять функциональность клиента, отправляя код клиенту в ответ на запрос. Это может быть полезно для динамического изменения поведения клиента.
Например, представьте, что у вас есть REST API для онлайн-магазина. Клиент может отправить запрос на сервер для получения информации о доступных товарах. В ответ сервер может вернуть код, например, JavaScript, который клиент должен выполнить для отображения списка товаров на своей веб-странице. Это позволяет серверу динамически расширять функциональность клиента, отправляя ему не только данные, но и код для их обработки и отображения.

# 5. что такое REST и SOAP
REST (Representational State Transfer) и SOAP (Simple Object Access Protocol) - это две различные методологии для обмена данными.

REST - это архитектурный стиль, который определяет принципы и ограничения для построения распределенных систем. Он использует протокол HTTP для обмена данными между клиентом и сервером. В REST API данные представлены в виде ресурсов, которые могут быть адресованы с помощью уникальных идентификаторов (URI). REST API обычно использует различные HTTP-методы, такие как GET, POST, PUT и DELETE, для выполнения операций над ресурсами. REST API также обеспечивает единообразный интерфейс, где данные могут быть представлены в различных форматах, таких как JSON или XML [0].

SOAP - это протокол обмена данными, который позволяет вызывать удаленные процедуры и передавать структурированные сообщения между клиентом и сервером. Он использует различные протоколы для передачи данных, такие как HTTP, SMTP и другие. SOAP-сообщения обычно представлены в формате XML и содержат информацию о вызываемом методе, параметрах и ожидаемом ответе. SOAP также обеспечивает механизмы для обработки ошибок и безопасности [1].

REST и SOAP имеют различные подходы к построению веб-сервисов. REST API является более простым и гибким, используя HTTP-протокол и единообразный интерфейс для обмена данными. SOAP, с другой стороны, предоставляет более мощные функции, такие как обработка ошибок и безопасность, но требует более сложной настройки и использования специальных библиотек для работы с SOAP-сообщениями.


- REST - это архитектурный стиль где REST предоставляет набор требований, которым нужно следовать, чтобы предоставить веб-службу RESTful. REST поддерживает несколько форматов файлов XML, JSON, TXT, CSV, HTML.
- REST это архитектруный стиль в котором описывается структура передачи информации. 
- SOAP - является протоколом обмена сообщниями, по которому веб-сервисы взаимодействуют друг с другом или с клиентом. для общения используется фалы в формате XML иначе сервер вернет ошибку. SOAP более безопасный чем REST. 
- отличия между SOAP and REST:
 rest использует разные форматы файла. soap только xml
 rest работает только с протоколами http/https. soap работает с разными протоколами
 rest может быть закэширован. soap нет
 rest это архитектурный стильно в котором нет огромного количества правил. soap это протокол который сильно ограничен теми правилами которые ему предъявляется
в случае rest это простота, скорость, расширяемость, нет привязки к правилам
в случае soap больше безопасности, и проще для тестирования, разработка дольше и энергозатратна

### 6. XML, JSON, HTML, CSS, JS file
**XML** (Extensible Markup Language) - используется для структурирования и хранения данных, представляет собой текстовый формат, который можно использовать для обмена информацией между клиентом и сервером.

**JSON** (JavaScript Object Notation) - это легкий формат для обмена данными, особенно популярен в веб-разработке. Поддерживает простую структуру данных, такую как текст, числа, логические значения и объекты, и используется для передачи данных между клиентом и сервером.

**HTML** (Hypertext Markup Language) - используется для создания структуры и содержимого веб-страницы. Описывает, как элементы должны быть отображены в браузере и позволяет добавлять текст, изображения, ссылки и другие элементы.

**CSS** (Cascading Style Sheets) - используется для задания внешнего вида веб-страницы, такого как цвета, шрифты, расположение элементов и т. д. CSS позволяет разработчикам легко изменять стиль и внешний вид веб-страницы без изменения ее структуры.

**JS** (JavaScript) - язык программирования, который выполняется в браузере и позволяет добавлять интерактивность и динамическое поведение на веб-страницах. JavaScript позволяет взаимодействовать с сервером, обрабатывать события пользователя и изменять содержимое страницы без перезагрузки.

Запрос HTTP состоит из трех основных компонентов: метода запроса, URI (Uniform Resource Identifier) и версии протокола.

Метод запроса определяет тип операции, которую клиентский агент (например, веб-браузер) хочет выполнить на сервере. Некоторые из наиболее распространенных методов запроса включают GET, POST, PUT и DELETE.

URI, также известный как URL (Uniform Resource Locator), указывает на конкретный ресурс, к которому клиент хочет получить доступ или совершить действие. URI включает доменное имя сервера и путь к ресурсу на сервере.

Версия протокола указывает на используемую версию протокола HTTP. Например, HTTP/1.1 или HTTP/2.

Кроме основных компонентов, запрос HTTP может также содержать заголовки запроса, которые передают дополнительную информацию о запросе, такую как тип контента, язык, кэширование и другие параметры. Заголовки могут использоваться для управления дополнительными аспектами запроса и обмена данными между клиентом и сервером.

### +9. из чего состоит ответ HTTP
- стартовая строка (http версия протокола http/1.1|статус код|статус текст)
- header заголовок
- body


---
Ответ HTTP (Hypertext Transfer Protocol) состоит из трех основных компонентов:

1. Стартовая строка (Start Line): Это первая строка ответа, которая содержит HTTP-версию сервера, статус ответа и соответствующее описание. Например:

   ```
   HTTP/1.1 200 OK
   ```

   В данном примере "HTTP/1.1" - версия HTTP, "200" - статус ответа (успешный код состояния), и "OK" - описание статуса.

2. Заголовки (Headers): Следующие строки ответа содержат заголовки, которые предоставляют дополнительную информацию о самом ответе, такую как тип контента, дата и время создания, сервер и другие заголовки. Примеры заголовков:

   ```plaintext
   Content-Type: text/html
   Date: Fri, 29 Nov 2019 12:00:00 GMT
   Server: Apache/2.4.29 (Unix)
   ```

3. Тело сообщения (Message Body): Опционально, тело сообщения содержит фактические данные, которые отправляются клиенту в ответ на его запрос. Тип содержимого определяется заголовком "Content-Type". Например, если "Content-Type" установлен как "text/html", то тело сообщения будет содержать HTML-разметку.

   ```html
   <!DOCTYPE html>
   <html>
   <head>
       <title>Пример страницы</title>
   </head>
   <body>
       <h1>Привет, мир!</h1>
   </body>
   </html>
   ```

Это общая структура ответа HTTP. Каждая часть имеет свое предназначение и содержит информацию, необходимую для правильной обработки ответа клиентом.

### ГОРЯЧИЙ\ХОЛОДНЫЙ РЕЗЕРВ СЕРВЕРА 
резервы сервера - нужны для обеспечения непрерывности работы важных компьютерных систем в случае сбоев или аварийных ситуаций.  
+ **Горячий резерв сервера** - это запасной компьютер, который всегда включен и готов сразу взять на себя работу, если основной сервер выходит из строя. Это обеспечивает быстрое восстановление работы.

+ **Холодный резерв сервера** - это также запасной компьютер, но он выключен и требует времени для включения и настройки, если основной сервер перестает работать. Это восстанавливает работу медленнее, чем горячий резерв.

### + 22. что такое КЭШ/КУКИ/ЛОКАЛ СТОРЕДЖ/СЕССИОН СТОРЕДЖ
- **cookies** это файлы создаваемые веб сайтами котоыре посещает юзер. в ней хранится информация об вашей атентификации, о предпочтениях и настройках юзера.
- **кэш** память содержит определенные файлы из сайта (изображения или видео). это надо чтоб в следующий раз сайт прогружался быстрее.
- **local storage** это веб хранилище где хранится информация ключ значение. информация обновляется после каждого обновления сраницы и даже после перезапуска браузера.
- **session storage** это веб хранилище которое сохраняет информацию в формате ключ значение, и обновляет информацию каждый раз при обновлении страницы. в случае закрытия вкладки браузера или полное закрытие браузера, данные блокируются.
>В браузерах существуют различные типы хранилищ данных, которые позволяют веб-приложениям и веб-сайтам сохранять информацию на компьютере пользователя. Вот некоторые из наиболее распространенных видов хранилищ браузера:

>Cookies (куки): Куки - это небольшие текстовые файлы, которые веб-сайты сохраняют на компьютере пользователя. Они могут содержать различную информацию, такую как данные аутентификации, предпочтения пользователя и другие данные. Куки обычно используются для отслеживания сеанса и хранения временных данных.

>LocalStorage (локальное хранилище): Локальное хранилище позволяет веб-приложениям сохранять данные в браузере пользователя без срока действия. Это полезно для хранения данных, которые должны сохраняться между сеансами браузера.

>SessionStorage (сеансовое хранилище): Сеансовое хранилище подобно локальному хранилищу, но данные в нем сохраняются только на время сеанса веб-пользователя. После закрытия браузера данные удаляются.

>IndexedDB: Это более мощная и гибкая система хранения данных, которая позволяет веб-приложениям сохранять структурированные данные в браузере. Это подходит для хранения больших объемов данных и обеспечивает более сложные запросы.

>Web Storage API: Web Storage API предоставляет доступ к локальному и сеансовому хранилищу. С его помощью можно легко сохранять и извлекать данные в браузере.

>Cache API: Cache API используется для кэширования ресурсов, таких как изображения, стили и скрипты. Он позволяет ускорить загрузку веб-страниц и сократить нагрузку на сервер.

>File System API: File System API предоставляет возможность веб-приложениям взаимодействовать с файловой системой компьютера пользователя. Однако его поддержка ограничена и не всегда доступна.

>Каждое из этих хранилищ имеет свои особенности и лучше подходит для определенных сценариев. Веб-разработчики выбирают наиболее подходящий тип хранилища в зависимости от требований своего проекта. Учтите, что использование хранилищ браузера должно соблюдать правила безопасности и уважать приватность пользователей.

### 25. как происходит авторизация на сервере
авторизация - это процесс, который позволяет клиенту войти на сервер и работать с ним, если он имеет право на это.

Когда клиент пытается подключиться к серверу, сервер проверяет, имеет ли клиент право это сделать. Это происходит так:

1 **Клиент спрашивает разрешение**: Клиент просит сервер разрешить ему войти.
2 **Сервер проверяет имя и пароль**: Если у клиента есть имя пользователя и пароль, сервер проверяет, совпадают ли они с теми, что он знает.
3 **Решение сервера**: Если проверка проходит успешно, сервер позволяет клиенту войти и делает доступными нужные данные.
4 **Работа с сервером**: Теперь клиент может взаимодействовать с сервером, запрашивая информацию или отправляя данные.
5 **Завершение сеанса**: Когда клиент закончил работу, сервер завершает соединение.

### 37. уровни OSI ПЕРЕСМОТРЕТЬ ТАК ЛИ ЭТО
-модель OSI (open system interconnection)  характеризует взаимодействие сетевого оборудования. простоыми словами описывает как работают сетевые устройства. она необходима для понимания как работают компьютерные сети. другими словами это процесс передачи данных который разделен на 7 уровней: 
1 уровень. физический, на нем происходит момент передачи физических сигналов,токов,импульсов. проще говоря это уровень проводов(блютуз,вайфай) на этом уровне рабоатют концетраторы и репиторы.
2 уровень. канальный. на этом уровне проверяется и исправляются ошибки передачи(информация об отправителе и получателе).
3 уровень. сетевой. на этом уровне происходит маршрутизация трафика. на этом уровне живет маршрутизатор
4 уровень. транспортный обеспечивает передачу данных по сети. он предназначен для надежной передачи информации от отправителя к получателю. тут обитают протоколы TCP and UDP. трафик чувствительный к потерям - TCP. если немного потеряем данных и это не стращно - UDP.
5 уровень. сеансовый. управляет соединением сеансами.
6 уровень. представления. происходит кодирование или сжатие файлов. преобразование.
7 уровень. приложения. этот уровень нужен чтобы приложения имели доступ к сетевым службам. доступ к файлам, пересылкам почты и запросам к базам данным.

### 38. TCP, TCP\IP, уровни
- TCP/IP в настоящий момент самый популярный набор протоколов TCP/IP, который задает стандарты связи между компьютерами и содержит подробные соглашения о маршрутизации и межсетевом взаимодействии. основвная функция данного набора это : электронная почта, передача файлов между компьютерами и удаленным выходом в сисетму.
- TCP выполняет проверку доставку всех файлов адресату и проверяет не было ли ошибки при передаче информации. TCP может повторно запросить данные если они были потеряны и регулировать загрузку сети. 
уровни : 1) канальный уровень. 2) межстековой уровень. 3) транспортный. 4) прикладной
- TCP/IP набор сетевых протоколов. передаваемый пакет проходит несколько уровней (как на почте: сначала пишем письмо, потом помещаем в конверт с адресом, затем на почте на нем ставится штамп и так далее.) 
- IP протокол это протокол сетевого уровня. задача этого уровня доставка IP пакетов от отправителя к получателю. кроме собственных данных, пакеты этого уровня имеют IP адрес отправителя и получателя. номера портов на сетевом уровне не используются, а также на этом уровне неизвестно, был ли пакет доставлен.
- TCP и UDP протоколы транспортного уровня. он находится над сетевым. на не к пакету добавляется порты отправителя и получателя. 
- TCP протокол с установлением соединения с гарантированной доставкой пакетов. если пакет не дошел, то он посылается повторно.
- UDP протокол без установления соединения и с негарантированной доставкой пакетов.
- над транспортным уровнем находмтся прикладной уровень, на котором работают такие протоколы как http, ftp и другие. например http и ftp используют надежный протокол TCP, DNS сервер работает через UDP.

- это кибер атака, при котором злоумышленники перехватывают разговор или передачу данных путем подслушивания, либо притворяясь его легальным участником. суть этой атаки кража конфиденциальных данных (банковские карты, номера, перехват платежных средств). по сколько это происходит в реальном времени, то такое отследить почти нереально. самое распространенное это делать ложную вайфай точку, через которую смогут украсть данные тех, кто подключился. самый лучший способ обезпаситьсебя, это заходить на сайты только httpS и использовать VPN
